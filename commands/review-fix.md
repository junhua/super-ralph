# Super-Ralph Review-Fix Command

Create a PR and launch an autonomous review-test-fix loop. Runs all regression tests (functional, integration, e2e) and uses pr-review-toolkit agents to find issues, repeating until no Critical or Important findings remain AND all tests pass. Then automatically merges the PR.

## Arguments

Parse the user's input for:
- **--max-iterations** (optional): Maximum review-fix cycles. Default: 0 (unlimited — loops until clean)
- **--aspects** (optional): Which review aspects to run. Options: `comments`, `tests`, `errors`, `types`, `code`, `simplify`, `all`. Default: `all`
- **--pr** (optional): Existing PR number to review. If omitted, creates a new PR.
- **--no-merge** (optional): Skip auto-merge after clean. Default: false (auto-merge enabled).

## Workflow

### Step 0: Create Isolated Worktree

Create a git worktree so the review-fix loop doesn't interfere with other Claude windows.

1. **Determine the target branch:**
   - If `--pr` provided: extract the branch from `gh pr view <PR_NUMBER> --json headRefName --jq '.headRefName'`
   - If no `--pr`: use current branch (`git branch --show-current`)

2. **Detect worktree directory** (autonomous — never ask the user):
   ```bash
   if [ -d .worktrees ]; then WTDIR=".worktrees"
   elif [ -d worktrees ]; then WTDIR="worktrees"
   else WTDIR=".worktrees"
   fi
   ```

3. **Ensure directory is git-ignored**:
   ```bash
   git check-ignore -q "$WTDIR" 2>/dev/null || echo "$WTDIR/" >> .gitignore
   ```

4. **Create worktree from the target branch:**
   ```bash
   SLUG="review-fix-$(echo "$BRANCH" | sed 's/[\/]/-/g')"
   git worktree add "$WTDIR/$SLUG" "$BRANCH"
   ```

5. **Install dependencies** (auto-detect):
   ```bash
   cd "$WTDIR/$SLUG"
   if [ -f bun.lock ] || [ -f bun.lockb ]; then bun install
   elif [ -f package.json ]; then npm install
   fi
   ```

6. **All subsequent steps run inside the worktree.**

7. **Report**: `"Worktree created at <path> for branch <BRANCH>. Starting review-fix."`

### Step 1: Ensure PR Exists

**If --pr provided:**
1. Verify PR exists: `gh pr view <PR_NUMBER> --json number,title,state`
2. If PR doesn't exist or is closed, report error and stop

**If --pr NOT provided:**
1. Detect current branch: `git branch --show-current`
2. Detect default branch: `gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'`
3. If on default branch, report error: "Cannot create PR from the default branch. Switch to a feature branch first."
4. Check for uncommitted changes: `git status --porcelain`
5. If uncommitted changes exist, warn but proceed (the review will catch them)
6. Push branch: `git push -u origin HEAD`
7. Create PR: `gh pr create --title "<auto-generated title>" --body "<auto-generated body>" --base <default-branch>`
8. Capture PR number from output

### Step 2: Load Review-Fix Skill

Invoke the `super-ralph:review-fix-loop` skill for the orchestration knowledge.

### Step 3: Construct Review-Fix Prompt

Read `${CLAUDE_PLUGIN_ROOT}/skills/review-fix-loop/references/review-fix-prompt.md` and fill in:
- `[PR_NUMBER]` → the PR number
- `[ASPECTS]` → the review aspects to run
- `[DEFAULT_BRANCH]` → the default branch name

### Step 4: Set Up Ralph Loop

Run the setup script:

```bash
${CLAUDE_PLUGIN_ROOT}/scripts/setup-ralph-loop.sh "<PROMPT>" --max-iterations <N> --completion-promise "REVIEW_CLEAN"
```

Where:
- `<PROMPT>` is the review-fix prompt from Step 3
- `<N>` is the max iterations (0 for unlimited)

### Step 5: Start First Iteration

Execute the first review-fix cycle:

1. **Run regression tests** — run ALL test suites (functional, integration, e2e):
   ```bash
   # Auto-detect runner
   if [ -f bun.lock ] || [ -f bun.lockb ]; then RUNNER="bun"
   elif [ -f package.json ]; then RUNNER="npm"
   fi
   $RUNNER test 2>&1
   ```
   Record results: total passed/failed/skipped, list of failing tests.
   Classify failures as NEW (caused by review-fix) or PRE-EXISTING.

2. **Run PR review agents** (dispatch via Task tool in parallel based on --aspects):
   - `code` → dispatch code-reviewer agent (include test results in context)
   - `errors` → dispatch silent-failure-hunter agent
   - `tests` → dispatch pr-test-analyzer agent
   - `comments` → dispatch comment-analyzer agent
   - `types` → dispatch type-design-analyzer agent (only if new types detected)
   - `simplify` → dispatch code-simplifier agent
   - `all` → dispatch all applicable agents

3. **Parse findings** — classify each issue by severity:
   - Critical (confidence >= 90): Must fix — includes new test failures
   - Important (confidence >= 80): Should fix
   - Minor: Log but don't fix
   - Suggestions: Log but don't fix

4. **Decision point:**
   - **No Critical AND no Important findings AND all tests pass** → output `<promise>REVIEW_CLEAN</promise>`, then proceed to merge
   - **Issues found OR tests failing** → dispatch issue-fixer agent with structured findings (including test failures), then commit and push

5. **Merge PR** (after REVIEW_CLEAN, unless `--no-merge` set):
   ```bash
   gh pr checks [PR_NUMBER] --watch   # wait for CI
   gh pr merge [PR_NUMBER] --squash --delete-branch
   ```

The ralph-loop Stop hook will handle subsequent iterations automatically.

## Review Agent Dispatch Templates

### Code Reviewer
```
Task tool:
  subagent_type: pr-review-toolkit:code-reviewer
  description: "Review PR #N for code quality"
  prompt: "Review the changes in this PR. Focus on bugs, logic errors, security vulnerabilities, and project convention violations. Only report issues with confidence >= 80."
```

### Silent Failure Hunter
```
Task tool:
  subagent_type: pr-review-toolkit:silent-failure-hunter
  description: "Hunt silent failures in PR #N"
  prompt: "Review the error handling in this PR's changes. Look for empty catch blocks, swallowed errors, inappropriate fallbacks, and missing error logging."
```

### Test Analyzer
```
Task tool:
  subagent_type: pr-review-toolkit:pr-test-analyzer
  description: "Analyze test coverage for PR #N"
  prompt: "Review the test coverage quality for this PR's changes. Identify critical gaps, missing edge cases, and tests that don't actually test real behavior."
```

## Issue-Fixer Dispatch Template

```
Task tool:
  subagent_type: super-ralph:issue-fixer
  description: "Fix review findings for PR #N"
  prompt: |
    Fix the following code review findings for PR #<PR_NUMBER>.

    ## Findings to Fix (Critical first, then Important)

    <PASTE STRUCTURED FINDINGS HERE>

    ## Instructions

    For each finding:
    1. Read the relevant code and understand context
    2. Implement the minimal fix
    3. Run tests: <PROJECT_TEST_COMMAND>
    4. Commit: git commit -m "fix: [what] (review-fix)"
    5. Push: git push

    Skip Minor and Suggestions — they don't block completion.
    NEVER ask for human input.
```

## Anti-Oscillation

Track fix history. If the same issue (same file, same type) appears in 3+ consecutive iterations:
1. This is oscillation — fixing one thing breaks another
2. Dispatch sme-brainstormer agents to analyze the root cause
3. The fix likely needs an architectural change, not a local patch
4. If sme-brainstormers can't resolve it, document in BLOCKED.md and output `<promise>REVIEW_BLOCKED</promise>`

## Critical Rules

- **Always create a worktree.** Never run a review-fix loop in the main working directory.
- **Never ask the user** about worktree location. Default to `.worktrees/` autonomously.
- **Run ALL regression tests every iteration.** Tests are the ground truth — code review alone is not enough.
- **Fix Critical before Important.** Severity order matters. New test failures are Critical.
- **Skip Minor and Suggestions.** They don't block completion.
- **Push after every fix commit.** The PR must reflect latest state for re-review.
- **Auto-merge after REVIEW_CLEAN.** Unless `--no-merge` is set. Use squash merge.
- **NEVER ask for human input.** Dispatch research-agent if stuck.
- **Track fix history.** Detect oscillation and escalate to architectural fix.
